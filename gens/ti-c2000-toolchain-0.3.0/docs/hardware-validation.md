# Hardware validation checklist

Offline tests prove protocol and conversion behavior but cannot prove probe,
flash-algorithm, reset or board-power behavior. Run this checklist for each board.

## Required inputs

- working `.ccxml` exported from CCS;
- exact CCS installation path;
- XDS probe model and serial number when multiple probes are connected;
- known-good `.out` executable and source;
- generated byte-addressed SVD;
- board boot-switch and power configuration.

## Test order

1. Open the `.ccxml` in CCS and verify connect, reset, load and run.
2. Install `extension/c2000-debug-0.3.0.vsix`.
3. Install `mcu-debug.peripheral-viewer`.
4. Copy the matching example launch configuration.
5. Start with `verifyProgram: false` and a simple RAM or flash blink program.
6. Verify halt, continue, restart and source stepping.
7. Set a source breakpoint in `main` and an instruction breakpoint at a known
   word-aligned address.
8. Inspect and modify `PC`, `ACC`, `XAR0`, `ST0` and `ST1` where supported.
9. Read a RAM region through VS Code's memory view and compare it with CCS.
10. Open the SVD peripheral tree and compare several registers with the CCS
    register view and the device technical reference manual.
11. Enable `verifyProgram: true` and repeat flash loading.
12. Save the adapter console output and bridge errors for any failure.

## F28M35x

Validate each core individually before trying the dual-core definition. Confirm
that the core regular expressions match exactly one CCS core each. Keep separate
executables and separate SVD files. The dual-core example uses the C28x SVD as the
primary viewer source; switch `svdPath` to the M3 SVD when inspecting the M3-owned
peripherals.
