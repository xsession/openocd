# Assessment Rubric

## Lab Notebook

| Level | Criteria |
| --- | --- |
| Excellent | Commands are reproducible, outputs are summarized, failures are explained, and hardware status is explicit. |
| Good | Commands and outputs are present with minor missing interpretation. |
| Developing | Some commands are present but context or result analysis is thin. |
| Needs work | Work cannot be reproduced from the submission. |

## Technical Understanding

| Level | Criteria |
| --- | --- |
| Excellent | Clearly separates adapter, transport, target, flash, server, and scripting responsibilities. |
| Good | Understands major OpenOCD layers with a few fuzzy boundaries. |
| Developing | Can run commands but struggles to explain what each layer does. |
| Needs work | Treats OpenOCD as a black box. |

## Safety And Hardware Discipline

| Level | Criteria |
| --- | --- |
| Excellent | Uses dummy/dry-run first, avoids destructive commands, and records hardware gates. |
| Good | Mostly safe with minor omissions. |
| Developing | Needs reminders before flash/security operations. |
| Needs work | Runs destructive commands without approval or evidence. |

## Capstone

| Level | Criteria |
| --- | --- |
| Excellent | Complete support package with configs or bridge, docs, metadata, validation, and known limits. |
| Good | Useful support package with small documentation or validation gaps. |
| Developing | Partial support package with unclear gates. |
| Needs work | No coherent integration story. |
