# Lab 23: Documentation And Release Notes

## Objective

Write user-facing docs and release notes for a support change.

## Safety

Documentation must not overstate support.

## Tasks

1. Choose a support lane from earlier labs.
2. Write a docs page with overview, commands, validation, and known limits.
3. Add an index entry.
4. Write a short release note.
5. Review for unsafe wording such as "full support" without flash and hardware
   evidence.

## Checkpoints

- The docs include exact commands.
- Known limits are visible.
- Hardware status is explicit.

## Deliverables

- User docs page
- Release note
- Known-limits checklist

## 30-Minute Session Plan

| Time | Activity |
| ---: | --- |
| 0-5 min | Review user-facing docs versus maintainer audit notes. |
| 5-12 min | Students outline a docs page for a chosen support lane. |
| 12-18 min | Add commands, validation evidence, and known limits. |
| 18-23 min | Draft a release note that does not overclaim support. |
| 23-27 min | Peer review for missing safety warnings or hardware status. |
| 27-30 min | Exit ticket. |

## Instructor Prompts

- What does a beginner need first?
- Where should destructive-command warnings appear?
- Why is "full support" a high bar?

## Exit Ticket

Write one release-note sentence that accurately describes partial support.
